## Context

The bicycle rental app runs locally via docker-compose (FastAPI backend + PostgreSQL). There is no cloud infrastructure. The CI pipeline builds and tests the app but has no deploy target. This design introduces the full AWS infrastructure needed for continuous deployment to staging and production, using Terraform IaC.

Constraints from AWS Academy LabRole:
- Cannot create IAM roles (`iam:CreateRole` denied) — must reference existing `LabRole` via data source
- Only `us-east-1` region available
- $50 budget cap — no multi-AZ RDS, no NAT Gateway, minimal Fargate sizing
- Session tokens expire in 1-4h — GitHub Actions secrets must be refreshed per session

## Goals / Non-Goals

**Goals:**
- Reproducible infra for staging and production via `terraform apply`
- ECS Fargate backend with ALB in each environment
- RDS PostgreSQL per environment, isolated credentials
- S3 static hosting for frontend, per environment
- Versioned S3 artifact bucket for frontend builds (prefix-based: `/v{sha}/`, `/stable/`)
- Runtime API URL injection via `window.ENV_API_URL` in `config.js`
- Alembic migrations as explicit one-off ECS task, not on app startup
- Shared Terraform state (S3 backend + DynamoDB lock table)

**Non-Goals:**
- HTTPS / TLS (LabRole cannot attach ACM certs to ALB listeners)
- CloudFront, custom domains
- Multi-AZ RDS or read replicas
- Autoscaling beyond single task
- NAT Gateway (Fargate tasks use public subnets for simplicity and cost)

## Decisions

### D1: ECS Fargate over ECS EC2 launch type

**Decision:** Use Fargate.

**Rationale:** No EC2 instance management, no ASG configuration, no AMI patching. The tutorial uses Fargate. For a CI/CD course project, Fargate removes operational noise and keeps Terraform focused on the deployment pipeline concepts.

**Alternative considered:** ECS EC2 launch type — would show EC2 instances in console but adds ASG, Launch Template, and instance IAM profile resources without educational benefit here.

### D2: Modular Terraform layout

```
infra/
├── modules/
│   ├── ecs-backend/     # ALB + ECS cluster + task def + service + security groups
│   ├── rds/             # RDS instance + subnet group + security group
│   ├── s3-frontend/     # S3 static website bucket + bucket policy
│   └── s3-artifact/     # S3 artifact bucket + versioning + lifecycle rules
├── envs/
│   ├── staging/         # calls modules, staging-specific vars
│   └── production/      # calls modules, production-specific vars
└── shared/              # artifact bucket (deployed once, shared across envs)
```

**Rationale:** Avoids copy-paste between staging and production. Each `envs/` root is a separate Terraform workspace with its own state. `shared/` is applied once (artifact bucket is environment-agnostic).

**Alternative considered:** Single flat `infra/` with `-var environment_name=staging` (like the tutorial) — simpler but doesn't separate state between environments, risking cross-environment blast radius.

### D3: Frontend API URL via runtime `config.js`

**Decision:** Build frontend once. Inject API URL at deploy time by uploading a `config.js` to each S3 environment bucket.

```js
// config.js (generated by CI, not committed)
window.ENV_API_URL = "http://staging-alb.elb.amazonaws.com";
```

`index.html` loads `<script src="/config.js"></script>` before the bundle. React code reads `window.ENV_API_URL || ""`.

**Rationale:** Building once avoids environment-specific build artifacts cluttering the artifact bucket. The `config.js` is cheap to generate and swap. Vite proxy stays intact for local dev.

**Alternative considered:** `VITE_API_URL` at build time — requires two separate builds, doubling artifact size and build time.

### D4: Frontend artifact versioning via S3 prefixes

```
s3://cicd-frontend-artifacts/
  v{GIT_SHA}/       ← immutable, uploaded once per commit
    index.html
    assets/
    config.js.template
  stable/           ← overwritten on each successful CI run
    (same files)
```

`stable/` is what CI promotes to environment buckets. Environments never pull directly from `v{sha}/` — CI always writes `stable/` first, then syncs `stable/` to the target bucket. This mirrors Docker's `latest`/`stable` tag pattern.

**Rationale:** Simple, transparent, no S3 Object Versioning API complexity. CI jobs can inspect `stable/` to know what's currently deployed without extra metadata stores.

### D5: Alembic migrations as separate ECS task

**Decision:** Remove `alembic upgrade head` from `entrypoint.sh`. Run migrations as a one-off ECS `run-task` call in CI before `ecs update-service`.

```yaml
# CI step order per environment:
1. terraform apply (ensure infra exists, get DB endpoint)
2. aws ecs run-task --task-definition migration-task (wait for exit 0)
3. aws ecs update-service --force-new-deployment
4. wait for service stable
5. run acceptance/smoke tests
```

The migration task definition uses the same Docker image as the app, overriding the command to `alembic upgrade head` only.

**Rationale:** Removes race condition when multiple app containers start simultaneously and each tries to migrate. Makes migration history explicit in CI logs.

**Alternative considered:** Keeping migrations in entrypoint — works for single-instance but breaks with >1 desired count and is invisible in CI.

### D6: Shared Terraform state (S3 + DynamoDB)

One S3 bucket and one DynamoDB table per environment root, plus one for `shared/`. Backend configured in each `envs/` root:

```hcl
backend "s3" {
  bucket         = var.tf_state_bucket   # injected by CI via -backend-config
  key            = "staging/terraform.tfstate"
  region         = "us-east-1"
  dynamodb_table = var.tf_lock_table
}
```

State bucket and lock table are created manually once (bootstrap step) since Terraform cannot create its own state backend.

### D7: RDS in public subnet, no NAT Gateway

**Decision:** Place RDS in the default VPC's public subnets but restrict security group to only accept connections from the ECS task security group.

**Rationale:** NAT Gateway costs ~$32/mo — exceeds LabRole budget for two environments. Public subnet with tight security group is standard for lab environments. Not appropriate for production systems beyond this course.

**Risk acknowledged:** RDS endpoint is technically reachable from internet if security group misconfigured. Mitigated by requiring SG source rule.

## Risks / Trade-offs

| Risk | Mitigation |
|------|------------|
| AWS Academy session tokens expire mid-pipeline | Document refresh procedure; CI step fails fast with clear error |
| `stable/` prefix overwritten mid-deploy (concurrent CI runs) | GitHub Actions concurrency groups per branch prevent parallel deploys |
| RDS in public subnet | Security group restricts access to ECS SG only; no public-facing DB port |
| Migration task fails silently | CI checks ECS task exit code before proceeding to service update |
| `config.js` cached by browser after env change | Set `Cache-Control: no-store` on `config.js` S3 object metadata |
| Single Fargate task = zero downtime only if new task starts before old stops | Set `minimumHealthyPercent: 100, maximumPercent: 200` in ECS service |

## Migration Plan

1. **Bootstrap** (manual, once): Create S3 state buckets + DynamoDB lock tables via AWS console or one-off CLI commands
2. **`terraform apply shared/`**: Create frontend artifact bucket
3. **`terraform apply envs/staging/`**: Create staging backend (ECS, ALB, RDS)
4. **`terraform apply envs/production/`**: Create production backend
5. **First CI run**: Build frontend → upload to artifact bucket → sync to staging S3 → run acceptance tests → sync to production S3 → smoke tests
6. **Verify**: Hit ALB URLs for backend health, S3 website URLs for frontend

**Rollback:** `terraform destroy` per env. For frontend: sync previous `v{sha}/` prefix back to environment bucket.
